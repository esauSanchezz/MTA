MasterTrackAgenda (MTA)

Base documental del sistema: alcance, arquitectura, modelo de datos, casos de uso y análisis de escalabilidad — pensada como referencia de desarrollo y como base de due-diligence si el producto se licencia o se vende a futuro.

01 · Producto

Resumen ejecutivo

Qué es MTA, a quién sirve y en qué punto está el desarrollo hoy.

MasterTrackAgenda (MTA) es una plataforma SaaS multi-tenant de gestión de negocios. Un solo backend y una sola base de código sirven a cuatro sectores verticales distintos — Belleza & Bienestar, Eventos Sociales, Distribución & Logística y Talleres & Servicios Técnicos — mediante un núcleo compartido (agenda, personal, ventas, inventario, reportes) más un módulo especializado por sector.

La propuesta de valor central es que agregar un sector nuevo no implica reescribir el sistema: se demostró al incorporar el sector Talleres reutilizando ~90% de las tablas ya existentes y añadiendo solo dos tablas nuevas (Assets, AssetRecords). Esto es lo que hace al producto atractivo como negocio — el costo marginal de atender un sector nuevo es bajo comparado con construir un sistema vertical desde cero.

02 · Producto

Alcance del sistema

Los cuatro sectores que cubre hoy, y los límites explícitos de la versión actual.
💆 Belleza & Bienestar

Consultorios clínicos de belleza, salones de belleza, barberías, salones de uñas. Agenda por servicio y empleado, expediente clínico del cliente (alergias, historial, fotos antes/después), POS y recibos.

🎉 Eventos Sociales

Salones de fiesta (incluye fiestas infantiles), bodas, XV años, eventos corporativos. Reservación simple con paquete, o contrato con anticipo, depósito de garantía y proveedores externos coordinados. Múltiples salones/espacios con disponibilidad independiente.

🚚 Distribución & Logística

Bodegas y mayoristas. Venta de mostrador sin reparto, o rutas de entrega (una o varias paradas) con vehículos e inventario en tránsito. GPS en tiempo real queda para Fase 2.

🔧 Talleres & Servicios Técnicos

Talleres mecánicos, de equipos y electrodomésticos. Ficha técnica del vehículo/equipo con historial de reparaciones, refacciones usadas y kilometraje/uso, en vez de expediente de persona.

Fuera de alcance en la versión actual

Ser honesto sobre esto ahora evita sorpresas después — sobre todo si el sistema se va a mostrar a un comprador o inversionista. Lo siguiente no está construido ni planeado dentro del roadmap de 4 meses:

  • Facturación fiscal electrónica (CFDI en México u otros esquemas fiscales por país) — hoy solo hay recibos internos en PDF, no comprobantes fiscales válidos ante autoridad.
  • Nómina y RRHH completo — se registra empleado, horario y comisión, pero no hay cálculo de nómina, prestaciones, IMSS/seguridad social, ni control de asistencia biométrico.
  • Contabilidad completa o integración contable (QuickBooks, Contpaqi, SAP, etc.) — el corte de caja da un resumen financiero, no un libro contable.
  • App móvil nativa — el frontend es una SPA web responsive (React), no hay app iOS/Android nativa todavía.
  • Multi-idioma y multi-moneda — el sistema está en español y opera en una sola moneda por instalación.
  • GPS en tiempo real — reservado como Fase 2 explícita para el sector Bodega; no está en el alcance de los 4 meses iniciales.
  • Franquicias o cadenas con reportes consolidados — cada Tenant es un negocio independiente; no hay una capa de "organización" que agrupe varios Tenants bajo un mismo dueño con reportes combinados.
  • E-commerce / tienda en línea de cara al cliente final — no hay un sitio de autoservicio donde el cliente compre o agende sin intervención de un empleado.
  • Transferencias de inventario entre bodegas — el modelo actual solo mueve inventario de la bodega al vehículo (manifiesto de ruta), no bodega-a-bodega.
  • API pública / marketplace de integraciones — no hay portal de desarrollador ni documentación de API abierta a terceros todavía.

04 · Ingeniería

Arquitectura y diseño técnico

El principio de diseño detrás de todo: núcleo compartido + módulo por sector.

MTA sigue una arquitectura en capas (Domain → Application → Infrastructure → API), con Azure Durable Functions como capa adicional para procesos asíncronos de larga duración. La multi-tenencia se resuelve con una única base de datos PostgreSQL y Global Query Filters de Entity Framework Core que agregan automáticamente WHERE TenantId = '...' a cada consulta — ningún controller necesita lógica extra para aislar los datos de cada negocio.

Decisiones de diseño clave y por qué

  • CQRS + MediatR: separa comandos (escritura) de consultas (lectura) por módulo. Esto es lo que permitió agregar el sector Talleres sin tocar el código de Belleza o Eventos — cada módulo vive en su propio conjunto de Commands/Queries.
  • PostgreSQL: costo bajo, buen soporte de tipos avanzados (JSON, arrays) útil si algún sector futuro necesita campos flexibles sin migrar todo el esquema.
  • Azure Durable Functions: los procesos largos o multi-paso (recordatorios de cita, corte de caja nocturno, reintentos de pago, onboarding) necesitan sobrevivir reinicios del servidor — una función normal con timeout no alcanza para esto.
  • JWT con TenantId embebido: el token de autenticación ya trae el TenantId, así que el Global Query Filter puede aplicarse sin una consulta extra a base de datos en cada request.
  • Patrón "núcleo + módulo por sector": el 90% de las tablas (Tenant, User, Employee, Customer, ScheduleTask, Service, Product, Sale, Receipt, Supplier, PurchaseOrder, CashRegisterCut) son compartidas; cada sector solo agrega 1-4 tablas propias. Esto es la base de la tesis de negocio: escalar a sectores nuevos es barato.
05 · Ingeniería

Modelo de datos

Vista conceptual: un núcleo compartido y cuatro módulos satélite por sector.

El detalle campo por campo de las 24 tablas (llaves primarias, foráneas, tipos) vive en el System Design Document — aquí el objetivo es mostrar la forma del modelo: qué es compartido y qué es específico de cada sector, para razonar sobre qué tan fácil es extenderlo.

Lectura clave para escalabilidad de producto: agregar un quinto sector significaría, en principio, sumar otra tabla satélite conectada al núcleo — no reescribir Tenant, User, Employee, ScheduleTask, Sale, ni ningún otro componente compartido. Ese aislamiento es intencional y es la razón por la que el patrón se pudo validar dos veces (Eventos y Talleres) sin fricción mayor.

06 · Ingeniería

Casos de uso

Los flujos principales, documentados como actor → objetivo → flujo → resultado.
UC-01
Agendar y cobrar una cita
Actor: Empleado
Precondición

El Customer existe o se crea en el momento; el empleado tiene un WorkSchedule vigente para el horario solicitado.

Flujo principal
  1. Se crea un ScheduleTask (Type=Appointment) vinculado a Customer y Service.
  2. Durable Function programa recordatorio 24h y 2h antes.
  3. El empleado marca InProgress → Completed.
  4. Se genera un Sale con SaleItems y se cobra (efectivo/tarjeta/transferencia).
  5. Se genera el Receipt en PDF y se envía por email.
Postcondición

Cita marcada como Completed, venta registrada, stock de productos usados descontado si aplica.

UC-02
Consultar expediente clínico antes de atender
Actor: Empleado
Precondición

El Customer ya tiene un ClinicalRecord de visitas anteriores.

Flujo principal
  1. Al abrir la cita agendada, el empleado consulta el ClinicalRecord del Customer.
  2. Revisa alergias y notas de tratamientos previos antes de atender.
  3. Al terminar, agrega nuevas notas y fotos antes/después al mismo ClinicalRecord.
Postcondición

Historial del cliente actualizado y disponible para su próxima visita.

UC-03
Contratar un evento con anticipo, garantía y proveedores
Actor: Owner / Cliente
Precondición

Existe al menos un Venue disponible para la fecha y aforo solicitado.

Flujo principal
  1. Se crea EventDetails (EventType, GuestCount) vinculado a un Venue sin traslape de horario.
  2. Se elige Mode=Contract: se define DepositAmount y SecurityDeposit.
  3. Se registran EventVendors (DJ, banquete, decoración) contra Suppliers existentes.
  4. Se cobra el anticipo (Sale parcial); más adelante, el saldo final el día del evento.
  5. Tras el evento, se revisa el espacio y se libera o retiene el SecurityDeposit.
Flujo alternativo

Mode=Simple: se salta contrato/garantía/proveedores y se reserva directo con un EventPackage.

Postcondición

Evento confirmado, Venue bloqueado para esa fecha/hora, pagos y garantía documentados.

UC-04
Planear y ejecutar una ruta de entrega
Actor: Chofer / Repartidor
Precondición

Hay un Vehicle disponible y Products con stock suficiente para cargar.

Flujo principal
  1. Se crea una DeliveryRoute con Employee, Vehicle y una o varias DeliveryStops.
  2. Se carga el manifiesto en VehicleInventory (QuantityLoaded por producto).
  3. Status=InProgress; en cada parada se descuenta QuantityDelivered y se genera un Sale.
  4. Se captura firma/foto como comprobante de entrega por parada.
  5. Al cerrar la ruta, se concilia QuantityLoaded vs. QuantityDelivered.
Postcondición

Ruta completada, ventas de cada parada registradas, inventario en tránsito reconciliado.

UC-05
Vender de mostrador, sin reparto
Actor: Empleado de mostrador
Flujo principal
  1. Cliente llega a la bodega; no se crea DeliveryRoute ni DeliveryStop.
  2. Se arma el Sale directo con SaleItems del catálogo de Products, Channel=Counter.
  3. Se cobra y se genera Receipt; el Stock general se descuenta igual que cualquier venta.
Postcondición

Venta registrada sin tocar ningún módulo de logística.

UC-06
Dar servicio a un vehículo/equipo con ficha técnica
Actor: Mecánico / Técnico
Flujo principal
  1. Se crea o reutiliza un Asset (vehículo/equipo) vinculado al Customer dueño.
  2. Se agenda un ScheduleTask Type=Service vinculado al Asset; el técnico revisa el AssetRecord previo.
  3. Se atiende el servicio y se registra un nuevo AssetRecord (diagnóstico, refacciones, kilometraje).
  4. Se cobra en POS (mano de obra + refacciones) y se genera el recibo.
  5. DF-08 programa un recordatorio de mantenimiento preventivo según kilometraje/tiempo.
Postcondición

Historial del activo actualizado; próximo mantenimiento programado automáticamente.

UC-07
Onboarding de un negocio nuevo (cualquier sector)
Actor: Owner nuevo
Flujo principal
  1. Se registra el Tenant con Sector, BusinessType, nombre y slug único.
  2. DF-04 carga catálogo default (servicios/paquetes, categorías de producto) según el Sector.
  3. Se crea el usuario Owner y se envía email de bienvenida.
  4. Si no inicia sesión en 3 días, se envía un recordatorio automático.
Postcondición

Negocio operativo sin intervención manual del equipo de MTA — el onboarding es 100% automático.

UC-08
Corte de caja diario automático
Actor: Sistema (Durable Function) → Owner
Flujo principal
  1. Timer diario (11:55pm) por Tenant activa DF-03.
  2. Se agregan todas las Sales del día (de cualquier sector/módulo) por método de pago.
  3. Se generan CashRegisterCuts con PDF y se envía por email al Owner.
Postcondición

El dueño recibe el cierre financiero del día sin pedirlo — cero trabajo manual de reporteo.

07 · Negocio

Escalabilidad

Qué tan lejos llega la arquitectura actual, qué se rompe primero, y qué tan escalable es como negocio.

Escalabilidad técnica

Hoy el sistema corre sobre una sola instancia de PostgreSQL compartida por todos los tenants, con aislamiento lógico vía Global Query Filters. Este diseño es simple, barato y funciona bien para un volumen inicial — pero tiene límites conocidos:

ComponenteRiesgo si crece sin cambiosMitigación
Base de datos única (PostgreSQL) Alto — un tenant con mucho volumen puede saturar la instancia y afectar a los demás ("noisy neighbor") Índices compuestos por TenantId, luego read replicas para reportes, y particionamiento/sharding por TenantId si se llega a miles de negocios
Consultas de reportes/dashboards Medio — sin caché, cada consulta pesada golpea la base directamente Capa de caché (Redis) para reportes y dashboards que no necesitan datos al segundo
API (ASP.NET Core en Azure App Service) Bajo — es stateless por diseño Escala horizontalmente agregando instancias; no requiere rediseño
Azure Durable Functions Bajo — autoescalable nativamente en Azure Monitorear si un sector (ej. Bodega con GPS en Fase 2) genera mucho más volumen de eventos que los demás, y aislar en un plan de consumo separado si es necesario
Azure Blob Storage (PDFs, fotos) Bajo — escala nativamente, costo lineal por uso Sin acción requerida a corto/mediano plazo
Conexiones a base de datos Medio — cada instancia de API abre su propio pool de conexiones Introducir PgBouncer (connection pooling) antes de escalar el número de instancias de API

En números aproximados: la arquitectura actual debería sostener sin cambios estructurales cientos de tenants pequeños/medianos con carga transaccional típica (agenda, POS, reportes diarios). El primer cuello de botella real aparece cuando (a) un solo tenant tiene un volumen de datos/transacciones desproporcionado, o (b) el número total de tenants concurrentes empieza a generar contención de conexiones en la base de datos. Ambos son resolubles sin rediseñar el modelo de datos — son cambios de infraestructura, no de arquitectura.

Escalabilidad de negocio (producto vendible)

  • Agregar un sector nuevo es barato: ya demostrado dos veces (Eventos, Talleres) sin tocar el núcleo. Un quinto sector es, en principio, 1-4 tablas nuevas más su módulo de Application/API.
  • Onboarding ya es automático (UC-07) — vender más tenants no implica más trabajo operativo de implementación por cliente.
  • Falta para vender como producto self-service: medición de uso por tenant (para planes de precio basados en volumen), un portal de facturación/suscripción, y documentación de API pública si se quiere permitir integraciones de terceros.
  • Modelo de precio sugerido: suscripción mensual por tenant, con el módulo de sector incluido según el tipo de negocio, y features como GPS en tiempo real (Fase 2) como add-on de pago una vez desarrollado.
  • Para una venta/adquisición del producto completo: lo que más valor agrega antes de esa conversación es (1) esta documentación misma, (2) métricas reales de uso/carga si ya hay tenants activos, y (3) un plan claro de qué falta para las brechas listadas en "Fuera de alcance."

Roadmap

Lo planeado a 4 meses, más lo que vendría después si el producto avanza hacia venderse.
Mes 1 — En curso (40%)

Backend Core

Estructura .NET, entidades de dominio, DbContext + JWT Auth, PostgreSQL en Docker, migración inicial, CRUD Empleados/Agenda.

Mes 2

Frontend React + Customers

Login/dashboard, calendario, gestión de empleados, inventario básico, Customers + ClinicalRecords + Assets/AssetRecords.

Mes 3

POS + Eventos + Azure

Ventas POS, Stripe Terminal, recibos PDF, EventDetails/EventPackages/EventVendors/Venues, Durable Functions principales.

Mes 4

Logística + Reportes + Polish

Vehicles/DeliveryRoutes/DeliveryStops/VehicleInventory, modo mostrador vs. ruta, reportes financieros, testing e2e.

Fase 2 — Post-lanzamiento

GPS en tiempo real

Tracking de vehículos en ruta, mapa en vivo, historial de recorrido. Requiere app móvil o dispositivo GPS — fuera del alcance inicial.

Fase 3 — Sugerida (no planeada aún)

Preparación para venta/escala como producto

Facturación fiscal electrónica por país, app móvil nativa, multi-idioma/multi-moneda, API pública, medición de uso por tenant, portal de auto-onboarding y facturación de suscripción.

09 · Referencia

Glosario y stack técnico

Términos clave y el stack completo, en un solo lugar.

Glosario

Tenant
Un negocio individual dentro del sistema multi-tenant. Todas las tablas cuelgan de un TenantId.
Multi-tenancy
Una sola instalación/base de datos sirve a muchos negocios distintos, con sus datos aislados entre sí.
Global Query Filter
Regla de Entity Framework Core que agrega automáticamente el filtro por TenantId a cada consulta, sin necesidad de repetirlo en cada controller.
CQRS
Command Query Responsibility Segregation — separar las operaciones de escritura (Commands) de las de lectura (Queries) en clases distintas.
MediatR
Librería de .NET que implementa el patrón mediador, usada para desacoplar Commands/Queries de los controllers.
Durable Function
Azure Function con estado, capaz de sobrevivir reinicios y coordinar procesos largos o multi-paso (ej. recordatorios, reintentos de pago).
Sector / BusinessType
Sector = una de las 4 verticales (Belleza, Eventos, Bodega, Talleres). BusinessType = el subtipo específico dentro del sector.
Núcleo compartido
Las 13 tablas y la lógica común a todos los sectores (agenda, personal, ventas, inventario, reportes).
